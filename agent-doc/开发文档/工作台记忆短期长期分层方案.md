# 工作台记忆短期长期分层方案

## 为什么做

工作台问答此前存在两个相互关联的问题。其一是记忆写入缺乏选择性的隐患：一旦开启对话自动记忆，容易把大量普通对话内容无差别地沉淀为长期记忆，长期下来会污染记忆库、抬高检索噪声、稀释真正有价值的稳定偏好。其二是长期记忆实际上处于"半离线"状态，`SystemSettingsService` 中虽然已经实现了本地记忆库的写入与检索能力，但工作台的对话主链路既没有在生成上下文时读取长期记忆，也没有在用户表达稳定偏好时写入长期记忆，导致用户在设置中心手工维护或期望系统自动沉淀的偏好、约束、目标，根本无法在后续问答中生效。

本方案的目标是按记忆的生命周期把记忆明确拆分为短期记忆与长期记忆两层，约束"什么内容值得记"，并把长期记忆真正接入工作台问答的读写主链路，使稳定偏好可以被沉淀、被召回、被遵循，同时严格避免把普通对话当作记忆无差别写入。

## 方案是什么

记忆按生命周期分为两层，各自承担不同职责，互不混用。

短期记忆承载单次会话内的过程状态，由 `ChatSessionState` 表达，包含当前会话的岗位列表、最近槽位、工具执行事件、简历匹配结果等。短期记忆随会话存在，生命周期与单个 `sessionId` 绑定，用于支撑同一轮对话内的上下文连续性与推理过程回看，不进入跨会话的长期记忆库。

长期记忆承载跨会话、跨任务的稳定语义信息，仅限稳定偏好、求职约束与求职目标这类高信号内容，存储在平台设置的本地记忆库（`settings.memory.items`）中，即使 `agent-memory` 服务不可用也能在本地降级生效。长期记忆的写入受双重约束：一是设置开关 `memory.enabled` 与 `memory.autoSaveChat`（界面文案为"自动保存稳定偏好"）控制是否允许自动沉淀；二是内容高信号判定 `shouldAutoWriteMemory` 控制只有包含"偏好、优先、排除、不要、不考虑、目标、期望、约束"等高信号词且长度达标的内容才被写入，普通对话内容一律不记录。长期记忆的读取受 `memory.enabled` 与 `memory.autoUseMemory`（界面文案为"按需使用相关记忆"）控制，且仅按当前问题做相关性检索，命中数量上限收敛为 2 条，避免噪声污染上下文。

## 具体怎么做

读取路径接入在个人上下文装配阶段完成。`PersonalContextBuilderImpl.build` 在判定当前问题需要个人上下文时（`needsPersonalContext`），调用 `SystemSettingsService.searchLocalMemories(message, 2)` 按当前问题召回高信号长期记忆，结果作为 `longTermMemory` 注入 `PersonalContext`。`PersonalContext` 新增 `longTermMemory` 字段，并在 `toMap()` 中以 `long_term_memory` 键透出、在 `sources()` 中追加"长期记忆"来源标识、在 `isEmpty()` 与摘要文案中纳入统计。由于 Runtime 侧 `ContextAssembler` 会把整个 `personal_context` 字典并入上下文摘要，新增的 `long_term_memory` 字段无需额外管线改造即可进入合成提示词。

写入路径接入在对话主链路完成。`ChatSseServiceImpl.handle` 在落库用户消息之后调用 `captureLongTermMemory(message)`，该方法先做轻量分类（包含"排除、不要、不考虑、约束"判为 `constraint`，其余高信号内容判为 `preference`），再转交 `SystemSettingsService.writeLocalMemory(type, content, "chat")`。真正的开关校验、高信号判定、去重合并、容量裁剪全部由 `SystemSettingsService` 统一负责，对话层只负责分类与转交，不重复实现记忆治理逻辑。

提示词侧，`agent-runtime/config/prompts/synthesis/default.md` 明确把 `long_term_memory` 列为 `personal_context` 中的一手证据，并要求合成器在回答与推荐时主动遵循其中记录的稳定偏好与约束。

## 涉及哪些模块和接口

后端 `agent-backend` 涉及 `PersonalContext`（新增 `long_term_memory` 字段与构造参数）、`PersonalContextBuilderImpl`（新增长期记忆检索装配）、`ChatSseServiceImpl`（新增高信号长期记忆写入与轻量分类，新增对 `SystemSettingsService` 的依赖注入）。记忆读写复用既有的 `SystemSettingsService.searchLocalMemories` 与 `SystemSettingsService.writeLocalMemory` 接口，未新增对外接口或 SSE 事件类型。Runtime 侧仅调整合成提示词文案，未改动接口契约。前端设置中心的记忆策略开关与记忆管理界面保持不变。

## 需要注意什么风险

第一，长期记忆写入默认依赖 `memory.autoSaveChat` 开关，默认关闭，因此默认不会自动写入；需要自动沉淀稳定偏好时应在设置中心开启该项，手工新增的记忆不受该开关影响。第二，高信号判定基于关键词，存在一定误判空间，但宁可漏记不可错记的取向能有效防止记忆库被普通对话污染。第三，长期记忆检索命中上限为 2 条且要求查询文本达到最小长度，避免短问题误召回与上下文膨胀。第四，记忆读写异常均被吞掉并降级为空，保证记忆能力故障不会阻断对话主链路。

## 如何验证

后端通过 `mvn -o clean compile` 验证编译通过，通过 `mvn test` 覆盖个人上下文装配与对话主链路。Runtime 通过 `uv run python -m pytest` 验证合成链路不回归。前端通过 `npm run build` 验证构建通过。端到端验证需在工作台开启"自动保存稳定偏好"后，发送一条包含明确偏好或约束的消息（例如"优先看上海大模型应用开发，排除外包驻场"），确认该记忆出现在设置中心记忆管理列表；随后在新会话中提出相关问题，确认回答能够遵循已沉淀的偏好与约束。

## 后续如何演进

后续可将长期记忆的写入判定从关键词规则升级为轻量分类模型，提高高信号识别准确率；可引入记忆的过期与冲突消解（离线 Dreams 流程），避免偏好随时间漂移产生矛盾；可在 `agent-memory` 服务可用时优先走服务端记忆库、本地记忆库作为降级路径，形成主备协同。
