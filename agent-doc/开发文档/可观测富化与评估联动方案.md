# 可观测富化与评估联动方案

## 为什么做

项目成熟度评估暴露出三个可观测短板。其一，Trace 事件缺少量化字段：`tool_execute_end` 只记录成功与否，无法回答"哪个工具慢、慢多少"；工具失败信息淹没在 end 事件里，没有独立的失败事件可供告警与评估订阅。其二，LLM 用量不可审计：`app/core/llm/usage.py` 已通过 contextvars 在运行期累计 token 用量，但这些数据只停留在响应 metrics 中，没有沉淀进 Trace，无法离线归因单次 run 的模型成本与缓存命中率。其三，HTTP 层缺少统一请求日志：跨模块联调时无法用 request_id 串联 Java 后端与 Runtime 的一次调用，问题定位依赖人工比对时间戳。同时，按仓库规约，Trace 节点与观测字段的变更必须同步 agent-eval 评分器与评估用例，否则新字段只是"写了没人看"的数据，无法进入质量门禁闭环。

## 方案是什么

在 agent-runtime 侧做三项增量富化，在 agent-eval 侧新增一个非破坏性的 observability 评分维度，两侧通过 Trace 事件契约衔接。

Runtime 侧：`TraceEventName` 新增 `tool_execute_failed` 与 `llm_usage` 两个事件。`tool_execute_end` 载荷扩展 `duration_ms`（本轮工具总耗时）与 `results`（逐工具的 `{tool, success, duration_ms}` 列表）；每个失败的工具调用额外产出一条 `tool_execute_failed` 事件，载荷含 `tool`、`error`、`duration_ms`、`permission_denied`。执行器在 run 结束（含成功、异常、流式四条路径）写 `run_end` 之前，若本次 run 发生过 LLM 调用，则记录一条 `llm_usage` 汇总事件，载荷为累计的 `llm_calls`、`prompt_tokens`、`completion_tokens`、`total_tokens`，流式路径附带 LLM 客户端的 Prompt 缓存指标 `llm_cache`。FastAPI 层新增 `request_logging_middleware`，透传或生成 `X-Request-Id`，记录方法、路径、状态码、耗时，`/health` 探活降为 debug 级别避免噪声。

Eval 侧：`grade_run` 新增 `_grade_observability_dimension`。该维度只在 trace 中出现相关事件或用例声明 `expect_llm_usage` 时产生检查项：`tool_execute_end` 出现时校验 `duration_ms` 与逐工具 `results` 结构；`tool_execute_failed` 出现时校验 `tool` 与 `error` 字段完备；`llm_usage` 出现时校验 `llm_calls` 与 `total_tokens`；用例声明 `expect_llm_usage: true` 时强制要求事件存在。旧 trace 不含新事件时该维度不产生任何检查项，`RUNTIME_REQUIRED_EVENTS` 保持不变，存量用例与评分结果完全不受影响。

## 具体怎么做

Runtime 改动集中在四个文件。`app/core/common/constants.py` 扩展 `TraceEventName` 枚举。`app/core/agent/graph.py` 的 `_execute_tool` 用 `_timed_execute` 包装网关调用计时，通过 `id(gateway_result)` 将耗时关联回结果对象（并行与串行两条执行路径共用同一关联方式），组装 `tool_stats` 后写入 end 事件载荷，并对失败项逐条落 `tool_execute_failed`。`app/core/agent/executor.py` 新增 `_record_llm_usage(trace_id, run_id, llm_client=None)` 辅助方法，读取 `current_usage()`，无 LLM 调用时直接返回不产生空事件；`execute` 与 `execute_stream` 的成功与异常路径均在 `run_end` 前调用。`app/server.py` 注册请求日志中间件。选择 run 级汇总而非逐调用事件，是因为 LLM 调用发生在多个异步节点内，逐调用记录会引入事件顺序与 done 事件采集的竞态，汇总事件则天然落在 `run_end` 之前的确定位置。

Eval 改动为 `agent-eval/app/grader.py` 新增维度函数并挂入 `grade_run`，`scripts/run_engine_eval.py` 的 `_grader_expected` 透传 `expect_llm_usage`，新增用例集 `cases/observability-v1.yaml`（LLM 问答路径、澄清路径两个安全用例，加一个标记 `requires_live_boss` 的工具耗时用例），新增测试 `tests/test_grader_observability.py` 覆盖正反载荷与向后兼容。

## 涉及模块与接口

agent-runtime：Trace 事件契约新增两个事件名、扩展一个事件载荷，属追加式变更，Java 后端与前端按事件名订阅的既有逻辑不受影响；HTTP 响应新增 `X-Request-Id` 头。agent-eval：`grade_run` 输出可能新增 `observability` 维度，`POST /v1/eval/run` 的响应结构不变。agent-frontend 与 agent-backend 无需改动。配套测试为 `agent-runtime/tests/test_observability_enrichment.py`（工具耗时载荷、失败事件、llm_usage 记录与跳过、中间件 request_id 透传）与 `agent-eval/tests/test_grader_observability.py`。

## 风险与注意事项

第一，`llm_usage` 依赖 contextvars 累计器，若未来引入跨任务的 LLM 调用（如后台预取），用量可能漏计，该事件应视为下界而非精确账单。第二，`id(gateway_result)` 关联依赖结果对象在收集与统计之间不被复制，若后续重构 `_execute_tool` 需保持该不变量或改用调用 id 关联。第三，评分维度对新事件是"出现即校验"，若 Runtime 后续调整载荷字段名，必须同步修改 grader 与用例，否则会产生系统性扣分。第四，请求日志不得输出请求体，避免敏感信息落日志。

## 如何验证

`agent-runtime` 执行 `uv run python -m pytest`（153 通过，含 5 个新观测测试）与 ruff 检查；`agent-eval` 执行 `uv run python -m pytest`（22 通过，含 8 个新维度测试）；前端补充的 composables/stores 单测执行 `npm test`（168 通过）。交付前运行 `./.agent-harness/scripts/gate.sh agent-runtime --quick` 与 `gate.sh agent-eval --quick`。在线验证可通过 `uv run python scripts/run_engine_eval.py --cases cases/observability-v1.yaml` 直连 Runtime 流式接口确认 `llm_usage` 事件真实产出。

## 增量二：日志上下文贯穿与服务间鉴权接入

在首轮富化落地后，日志层仍存在链路字段断层：请求入口有 request_id、执行入口有 run_id，但 graph 节点、工具网关、LLM 客户端等嵌套模块产生的日志不携带任何链路字段，跨模块排障仍需人工比对时间戳。本增量通过 loguru 的 contextvars 机制补齐该断层。新增 `app/core/common/logging.py` 统一配置 sink，通过 patcher 将 `logger.contextualize` 注入的全部 extra 字段渲染进日志行；`create_app` 启动时调用 `setup_logging`，日志级别由环境变量 `LOG_LEVEL` 控制。请求日志中间件将整个请求处理链包在 `logger.contextualize(request_id=...)` 内；`AgentExecutor.execute` 与 `execute_stream` 拆分为薄壳与内部实现两层，薄壳生成 run_id、session_id、trace_id 后以 `logger.contextualize` 包裹内部实现，使整次 run 内所有嵌套模块日志自动携带链路字段，各处日志文案不再需要手工拼接这些 ID。同时将此前孤立的 `app/internal_auth.py` 接入 `create_app`：注册顺序上先装鉴权中间件、后装请求日志中间件，保证 401 拒绝同样留下访问日志；`AGENT_INTERNAL_SERVICE_TOKEN` 未配置时鉴权不生效，保持本地联调便利，该变量与 `LOG_LEVEL` 已补入根目录 `.env.example`。需要注意的是，若在生产启用该令牌，agent-backend 及其他调用方必须同步注入 `X-Internal-Service-Token` 请求头，目前 Java 侧尚未实现该注入，属后续工作。配套测试为 `tests/test_logging_context.py`（上下文渲染、无上下文纯净输出、请求与 run 双层上下文合并）与 `tests/test_internal_auth.py`（未配置放行、缺失与错误令牌拒绝、正确令牌放行、/health 豁免）。本增量不改变任何 Trace 事件名与载荷结构，agent-eval 评分器与用例无需同步修改。

## 后续演进

短期可将 `llm_usage` 汇总扩展为按节点分桶（理解、规划、合成各自的 token 占比），支撑 Prompt 分层优化的量化依据；中期把 `duration_ms` 与缓存命中率接入 OTel 指标导出，与既有的 `Runtime Trace接入OpenTelemetry方案` 汇合，形成面板化的成本与时延监控；评估侧可在 observability 维度上叠加预算类断言（如单 run token 上限），把成本回归纳入质量门禁。
